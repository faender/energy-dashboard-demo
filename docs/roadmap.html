Was pro Quartal passiert

Q1 — Fundament

Wissen vor Code

  • Bestandsaufnahme: welche Hersteller/Protokolle wirklich im Portfolio (OPC-UA, Modbus, proprietäre BMS-APIs)
  • Architekturentscheidung: Zeitreihen-DB, Cloud vs. Edge/On-Prem
  • Ein Pilotstandort wird vertraglich und technisch vorbereitet
aus der Demo → SQLite wird durch TimescaleDB/InfluxDB ersetzt
Q2 — Integration

Die echte Ingestion

  • Echter OPC-UA-/Modbus-Client für den Pilotstandort
  • Reconnect-Logik, Datenqualitäts-Codes, Backpressure
  • Alarmlogik auf gleitende Zeitfenster umstellen
aus der Demo → nur scada_client.py + alarms.py werden ausgetauscht, Rest bleibt
Q3 — Pilot

Am echten Standort beweisen

  • Live-Betrieb mit echten Technikern, Alarme gegen reale Störungen validieren
  • Rollen/Mandantenfähigkeit, Anbindung an bestehendes Wartungssystem (CMMS)
  • Benachrichtigungen (E-Mail/SMS) für kritische Alarme
aus der Demo → Wartungstickets werden an SAP PM/Maximo statt an eine eigene Tabelle angebunden
Q4 — Rollout

Vom Pilot zum Portfolio

  • Schrittweiser Anschluss aller Standorte, Redundanz/HA für die Integrationsschicht
  • Security-Review, Dokumentation, Schulung, SLA-Definition
  • Erstes Predictive-Maintenance-Modell auf gesammelten Felddaten
aus der Demo → predictive.py-Heuristik wird durch trainiertes Modell ersetzt

Risiken & offene Fragen

Zugriff auf Hersteller-SCADA — bestehende Wartungsverträge garantieren nicht automatisch API-/OPC-UA-Zugang. Muss vertraglich geklärt werden, bevor Q2 startet. hoch
Regulatorik kritischer Infrastruktur (NIS2 u.ä.) kann Auth-, Logging- und Audit-Anforderungen nachträglich verschärfen und den Zeitplan verschieben. hoch
Protokoll-Vielfalt im echten Portfolio unbekannt — je mehr Hersteller, desto mehr eigene Clients statt eines gemeinsamen Gateways (Kepware/Ignition). mittel
Konnektivität an entlegenen Standorten — evtl. Edge-Gateway mit Zwischenspeicherung nötig, falls Cloud-Anbindung nicht durchgehend verlässlich ist. mittel
