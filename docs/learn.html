Sprache & Typisierung

Type Hints — und was sie NICHT tun

Python selbst prüft Type Hints zur Laufzeit nicht. def wind_power_kw(asset: Asset, ws: float) -> float: ist erstmal nur Dokumentation — rufst du die Funktion mit einem String statt einem Asset-Objekt auf, crasht an dieser Zeile nichts. Der Fehler zeigt sich erst später, wenn der Code versucht, z.B. asset.rated_power_kw zu lesen und feststellt, dass ein String kein solches Attribut hat.

Type Hints entfalten ihre eigentliche Kraft erst durch Werkzeuge, die sie aktiv auslesen: dein Editor (Autovervollständigung, "meinst du wirklich diesen Typ?"), mypy/pyright (statische Prüfung ohne den Code auszuführen) — und in diesem Projekt vor allem Pydantic und FastAPI, die Type Hints zur Laufzeit benutzen, um ankommende Daten tatsächlich zu validieren und zu konvertieren (siehe Pydantic-Karte weiter unten).

überall im Backend, z.B. simulation.py

from __future__ import annotations — warum diese Zeile überall steht

Normalerweise wertet Python eine Typannotation im selben Moment aus, in dem Funktion oder Klasse definiert wird. Bei def f(x: Foo) -> Bar: muss Foo und Bar zu diesem Zeitpunkt im Modul bereits existieren — sonst ein NameError. Das wird lästig bei Selbstreferenzen oder Verweisen auf eine erst weiter unten definierte Klasse.

from __future__ import annotations (PEP 563) ändert das grundlegend: jede Annotation wird nur noch als reiner, unausgewerteter Text gespeichert. Erst wenn eine Bibliothek wie Pydantic oder SQLAlchemy die Annotation tatsächlich braucht, liest sie sie gezielt aus — zu einem Zeitpunkt, an dem garantiert alle Namen existieren. Praktischer Nebeneffekt in diesem Projekt: die kurze X | None-Schreibweise und eingebaute generische Typen wie list[dict] funktionieren dadurch unabhängig von der genauen Python-Version.

@dataclass — was du sonst von Hand schreiben müsstest

Ohne @dataclass müsste Asset in portfolio.py so aussehen:

class Asset:
    def __init__(self, site_id, asset_id, type, name, rated_power_kw,
                 rotor_diameter_m=None, cut_in_ms=3.0, ...):
        self.site_id = site_id
        self.asset_id = asset_id
        # ... eine Zeile pro Feld, von Hand
    def __repr__(self):
        return f"Asset(site_id={self.site_id!r}, asset_id={self.asset_id!r}, ...)"
    def __eq__(self, other):
        return isinstance(other, Asset) and self.site_id == other.site_id and ...

@dataclass generiert __init__, eine lesbare __repr__ (wichtig fürs Debuggen — print(asset) zeigt alle Felder) und __eq__ (zwei Asset-Instanzen mit denselben Feldwerten gelten als gleich) automatisch aus den reinen Feld-Deklarationen.

seed: int = field(default=0) ist hier austauschbar mit = 0, weil int unveränderlich ist. field() wird zwingend, sobald ein Default veränderlich ist: x: list = [] würde dieselbe Listeninstanz für jede einzelne Asset-Instanz wiederverwenden (ein klassischer Python-Fallstrick) — field(default_factory=list) erzeugt dagegen bei jeder neuen Instanz eine frische Liste.

class AssetType(str, Enum) — doppelte Vererbung mit Zweck

Diese Klasse erbt gleichzeitig von str und von Enum. Das Ergebnis verhält sich in jeder Hinsicht wie ein String (AssetType.WIND_TURBINE == "wind_turbine" ist True, er lässt sich direkt in JSON schreiben), ist aber gleichzeitig ein abgeschlossener, benannter Wertebereich — Editor und Typchecker wissen, dass hier nur genau drei Werte möglich sind, nicht "irgendein String".

Ein reines Enum (ohne str) würde sich beim Serialisieren als "AssetType.WIND_TURBINE" (der Name, nicht der Wert) zeigen, sofern man nicht überall explizit .value anhängt. Die doppelte Vererbung erspart genau diesen Schritt an jeder Stelle, an der FastAPI/Pydantic den Wert automatisch in eine Antwort schreibt.

Dict als Lookup-Tabelle statt if/elif-Kette

Die Alternative zu RAW_TAG_TO_METRIC wäre eine Kette wie if tag_name == "active_power_kw": metric = "power_kw" elif tag_name == "power_kw": metric = "power_kw" elif ... — funktional gleichwertig, aber jede neue SCADA-Tag-Variante würde eine weitere elif-Zeile innerhalb der Funktionslogik bedeuten, mit Risiko, eine Bedingung falsch zu treffen oder eine Reihenfolge-Abhängigkeit einzubauen.

Als Dict ist die Zuordnung reine Daten: RAW_TAG_TO_METRIC.get(tag_name) ist ein einziger, immer gleicher Lookup — ein neuer Tag bedeutet "eine Zeile in einer Tabelle ergänzen", nicht "die Funktion normalize_tag_readings anfassen".

normalize.py Zeile 15

defaultdict(list) — was ohne sie nötig wäre

Ein normales Dict wirft bei d[key].append(x) einen KeyError, wenn key noch nicht existiert. Ohne defaultdict müsste man vor jedem Anhängen schreiben: if key not in d: d[key] = [] — dann erst d[key].append(x).

defaultdict(list) ruft bei jedem ersten Zugriff auf einen unbekannten Key automatisch list() auf und legt das Ergebnis (eine leere Liste) als Wert ab, bevor .append() überhaupt ausgeführt wird — die Prüfung verschwindet komplett aus dem Aufrufer-Code.

kpi.py — _fetch_metric(), yield_series()

zip(points, points[1:]) — Paare aus Nachbarn bilden

Ein konkretes Durchrechnen: aus points = [A, B, C, D] wird points[1:] zu [B, C, D]. zip kombiniert beide Listen Index-für-Index und stoppt bei der kürzeren: (A,B), (B,C), (C,D). Jedes Element außer dem allerersten und dem allerletzten taucht also genau zweimal auf — einmal als "zweiter Partner", einmal als "erster Partner" des nächsten Paares.

Für die Trapez-Integration in kpi.py ist das exakt das, was gebraucht wird: jeweils zwei zeitlich benachbarte (Zeitpunkt, Leistung)-Messungen, um die Fläche des Trapezes zwischen ihnen zu berechnen (siehe Walkthrough bei kpi.py weiter unten).

Async & nebenläufiges I/O

async def / await — warum blockierendes I/O hier ein Problem wäre

Ein normaler, synchroner HTTP-Aufruf (z.B. mit requests.get(url)) blockiert den kompletten Python-Prozess, bis die Antwort da ist — in dieser Wartezeit kann derselbe Prozess absolut nichts anderes tun, auch keinen eingehenden Dashboard-Request beantworten.

async def + await funktionieren kooperativ: trifft eine await-Zeile auf eine noch nicht fertige I/O-Operation (hier: eine HTTP-Antwort vom Mock-SCADA), gibt die Funktion die Kontrolle an den Event-Loop zurück. Der Event-Loop nutzt genau diese Lücke, um z.B. einen wartenden Dashboard-Request zu bedienen, und kehrt erst zur ursprünglichen Funktion zurück, sobald deren Antwort da ist. Das ist der Grund, warum ein einziger integration-service-Prozess gleichzeitig die REST-API bedienen und im Hintergrund alle 15 Sekunden pollen kann — ohne Threads, ohne mehrere Prozesse.

scada_client.py, ingestion.py, main.py

Async Context Manager (async with) — Aufräumen wird garantiert

Ein Context Manager garantiert "Aufräumen", egal ob der Codeblock normal durchläuft oder eine Exception wirft — bekannt von with open(file) as f:, das die Datei garantiert schließt. async with ist dieselbe Idee für Ressourcen, deren Öffnen/Schließen selbst await-fähig ist.

async def fetch_live_readings() -> list[dict]:
    async with httpx.AsyncClient(base_url=SCADA_BASE_URL, timeout=30) as client:
        resp = await client.get("/scada/live")
        resp.raise_for_status()
        return resp.json()
    # client wird HIER garantiert geschlossen — auch wenn raise_for_status()
    # wegen eines 500ers eine Exception wirft

Endlosschleife mit asyncio.sleep — die einfachste Lösung, bewusst

while True: await poll_once(); await asyncio.sleep(15) ist der einfachste denkbare Weg für einen wiederkehrenden Hintergrundjob — kein Cron, kein Celery, kein externer Scheduler-Prozess. Der await in asyncio.sleep() gibt (genau wie ein HTTP-Call) die Kontrolle an den Event-Loop zurück und blockiert daher nicht den Rest der Anwendung während der 15 Sekunden Wartezeit.

Grenze dieser einfachen LösungLiefe integration-service mehrfach (mehrere Container-Replicas für Ausfallsicherheit), würde jede Instanz unabhängig und parallel pollen — für eine Demo mit genau einer Instanz kein Problem, für einen horizontal skalierten Produktivbetrieb bräuchte man Koordination (z.B. nur eine aktive Instanz pollt, oder ein externer Scheduler übernimmt das Timing).
ingestion.py — poll_live_forever()

@asynccontextmanager für den App-Lebenszyklus — Schritt für Schritt

@asynccontextmanager verwandelt eine ganz normale async def-Generatorfunktion mit genau einem yield in etwas, das FastAPI direkt als lifespan-Parameter akzeptiert.

@asynccontextmanager
async def lifespan(app: FastAPI):
    Base.metadata.create_all(bind=engine)   # 1. Tabellen anlegen, falls nicht vorhanden
    await backfill_history()                # 2. einmalig warten, bis 30 Tage Historie da sind
    _poll_task = asyncio.create_task(poll_live_forever())  # 3. Hintergrund-Task STARTEN, nicht abwarten
    yield                                    # 4. --- App läuft, bedient Requests ---
    _poll_task.cancel()                      # 5. beim Herunterfahren sauber abbrechen

Entscheidend ist Zeile 3: asyncio.create_task(...) ohne await davor. Mit await würde die Startsequenz für immer an dieser Stelle hängen bleiben, weil poll_live_forever() nie zurückkehrt (Endlosschleife) — die App käme nie bei yield an und würde nie anfangen, Requests zu bedienen. create_task stößt die Endlosschleife nur an und lässt sie parallel im Hintergrund weiterlaufen, während die Startsequenz sofort zu yield weiterspringt.

Backend-Frameworks (FastAPI · SQLAlchemy · Pydantic)

Pydantic BaseModel — Validierung, nicht nur Struktur

Eine Pydantic-Klasse wie AssetOut(BaseModel) sieht einem @dataclass ähnlich, tut aber beim Erzeugen einer Instanz aktiv etwas: sie validiert und konvertiert. Bekommt ein als float deklariertes Feld den String "3.5" übergeben, wandelt Pydantic das automatisch in 3.5 um — oder wirft einen klaren Validierungsfehler, wenn das nicht möglich ist.

FastAPI nutzt das doppelt: für eingehende Request-Parameter (automatische, aussagekräftige 422-Fehler bei falschem Typ, ganz ohne eigenen Prüfcode) und für response_model=... — die zurückgegebene Antwort wird auf das Schema zugeschnitten, Felder, die im ORM-Objekt zusätzlich existieren, aber nicht im Schema stehen, verschwinden aus der JSON-Antwort. Als Nebeneffekt entsteht daraus automatisch die interaktive Doku unter /docs.

schemas.py, scada-simulator/app/models.py

SQLAlchemy 2.0 Declarative Mapping — eine Klasse, zwei Bedeutungen

asset_id: Mapped[str] = mapped_column(String, primary_key=True) sagt zwei Dinge gleichzeitig: Python-seitig ist asset.asset_id ein str (Editor/Typchecker wissen das über Mapped[str]). DB-seitig wird daraus beim Anlegen der Tabelle eine Spalte asset_id VARCHAR PRIMARY KEY (über mapped_column(String, primary_key=True)).

Diese doppelte Information an einer Stelle ist der Kern eines ORM: man schreibt Python-Klassen, SQLAlchemy übersetzt sie in CREATE TABLE, und jede spätere Objektänderung (asset.name = "neu"; session.commit()) automatisch in UPDATE — ohne dass irgendwo im restlichen Code SQL-Text vorkommt.

Decorator-Routing (@app.get(...)) — was ein Decorator wirklich tut

Ein Decorator ist nur eine Funktion, die eine andere Funktion entgegennimmt und (meist) eine registrierte oder veränderte Version zurückgibt. @app.get("/health") über def health(): ist gleichbedeutend mit health = app.get("/health")(health).

app.get(...) trägt die Funktion in eine interne Routing-Tabelle ein (Pfad → Handler) und gibt sie unverändert zurück — man kann health() trotzdem ganz normal wie eine Python-Funktion aufrufen und in einem Test direkt testen, ganz ohne laufenden Server.

alle main.py-Dateien

Session-per-Request — warum nicht eine geteilte Verbindung?

Jede Interaktion mit der Datenbank — ein Dashboard-Request, ein Backfill-Durchlauf, ein einzelner Live-Poll — holt sich am Anfang eine frische Session über get_session()/SessionLocal() und schließt sie in einem finally-Block, egal ob der Code davor erfolgreich durchlief oder eine Exception warf.

Eine Session ist mehr als eine offene Verbindung: sie bildet eine Transaktionsgrenze (Änderungen bis zum commit() gehören zusammen oder werden gemeinsam verworfen) und einen Identity-Cache. Eine einzige, dauerhaft geteilte Session über alle Requests hinweg würde diese Isolation aufheben — ein halbfertiger Request könnte Änderungen eines anderen, noch nicht commiteten Requests sehen.

integration-service/app/main.py, ingestion.py

Idempotentes Schreiben (ON CONFLICT DO NOTHING)

UniqueConstraint("asset_id", "metric", "timestamp") in models.py erzwingt auf DB-Ebene: für dieselbe Anlage/Metrik/Zeitstempel-Kombination kann es nie zwei Zeilen geben. sqlite_insert(Reading).values(rows).on_conflict_do_nothing(index_elements=[...]) sagt der Datenbank: versuche das Insert, und falls genau dieser Constraint verletzt würde, ignoriere den Konflikt still, statt einen Fehler zu werfen.

Das erspart eine explizite "existiert das schon?"-Prüfung im Python-Code vor jedem Schreiben — wichtig, weil sich Backfill (bis Zeitpunkt X) und der allererste Live-Poll (direkt danach) im Zeitfenster leicht überlappen können.

Deterministischer Zufall über Hashing — die Mathematik dahinter

hashlib.sha256(key).hexdigest() erzeugt aus einem beliebigen Eingabe-String (hier: Anlagen-ID + Zeitstempel + ein Label wie "gust", mit | verbunden) einen 64-stelligen Hex-String, der bei der kleinsten Änderung der Eingabe komplett anders aussieht, bei identischer Eingabe aber immer identisch ist.

def _stable_random(*parts: object) -> float:
    key = "|".join(str(p) for p in parts).encode()
    digest = hashlib.sha256(key).hexdigest()
    return int(digest[:12], 16) / 16**12

int(digest[:12], 16) liest die ersten 12 Hex-Zeichen als Ganzzahl zur Basis 16 (Werte von 0 bis 16¹²-1). Die Division durch 16**12 normiert das Ergebnis auf einen Wert zwischen 0 und 1 — im Effekt eine "Zufallszahl", die reproduzierbar ist. Genau das erlaubt /scada/live (jetzt) und /scada/history (rückwirkend für denselben Zeitpunkt) exakt konsistente Werte zu liefern, ohne dass irgendwo eine Zeitreihe gespeichert werden müsste — der "Speicher" steckt vollständig in der mathematischen Funktion, nicht in einer Datenbank.

TypeScript / React

TS interface als von Hand gepflegter Vertrag

Ein TypeScript interface beschreibt nur die Form von Daten und existiert zur Laufzeit gar nicht mehr — reine Compile-Zeit-Information für Compiler und Editor. interface Asset { ...; status: "online" | "offline" | "maintenance" | "fault"; } in api/client.ts ist von Hand so geschrieben, dass es exakt der Pydantic-Klasse AssetOut in schemas.py entspricht — inklusive der exakten String-Literale für status (ein "String-Literal-Union-Typ", der genau diese vier Werte erlaubt).

Was bei Drift passiertÄndert das Backend ein Feld, ohne dass jemand client.ts nachzieht, merkt TypeScript das nicht — anders als bei Pydantic ist diese Prüfung rein statisch und endet an der Backend-Grenze. Generische Funktion get<T>() — wie TypeScript T ableitet

T ist ein Platzhalter für einen Typ, den TypeScript aus der Verwendung ableitet, nicht aus der Definition. Bei get<PortfolioSummary>("/api/portfolio/summary") setzt der Aufrufer T = PortfolioSummary explizit.

async function get(path: string, params?: ...): Promise {
  const res = await fetch(url.toString());
  if (!res.ok) throw new Error(`API-Fehler ${res.status} bei ${path}`);
  return res.json();  // res.json() liefert eigentlich "any" — die Promise<T>-Signatur
  // BEHAUPTET nur, dass es ein T ist, ohne das zur Laufzeit zu prüfen
}

Diese Behauptung wird — anders als bei Pydantic im Backend — nicht zur Laufzeit geprüft. Ein Feld, das das Backend umbenennt, würde hier erst als undefined im UI auffallen, nicht als Fehler beim Fetch selbst.

Custom Hook usePolling — Zeile für Zeile

export function usePolling(fetcher: () => Promise, intervalMs: number, deps: unknown[] = []) {
  const [data, setData] = useState(null);
  const [error, setError] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    let cancelled = false;          // 1: Flag für DIESEN Effekt-Durchlauf
    setLoading(true);

    async function tick() {
      try {
        const result = await fetcher();
        if (!cancelled) { setData(result); setError(null); }  // 2: nur setzen, wenn noch aktuell
      } catch (err) {
        if (!cancelled) setError(err as Error);
      } finally {
        if (!cancelled) setLoading(false);
      }
    }

    tick();                          // 3: sofort einmal ausführen
    const id = setInterval(tick, intervalMs);  // 4: danach im Intervall wiederholen
    return () => { cancelled = true; clearInterval(id); };  // 5: Cleanup bei Unmount/Deps-Wechsel
  }, deps);

  return { data, error, loading };
}

Zeile 5 ist der wichtigste Teil: React ruft diese Cleanup-Funktion automatisch auf, bevor ein neuer Effekt-Durchlauf beginnt (weil sich ein Wert in deps geändert hat) oder wenn die Komponente ganz verschwindet. Ohne cancelled würde clearInterval allein nur zukünftige Ticks stoppen — ein bereits laufender fetch würde trotzdem irgendwann seine Antwort zurückbekommen und versuchen, setData aufzurufen.

lib/usePolling.ts

cancelled-Flag — das konkrete Fehlerszenario

Stell dir vor, im Service-Dashboard wechselt der Nutzer den Anlagentyp-Filter zweimal schnell hintereinander: erst "Windkraft", dann sofort "Photovoltaik". Für "Windkraft" läuft bereits ein fetch, während für "Photovoltaik" (neuer useEffect-Durchlauf, weil sich assetType in deps geändert hat) ein zweiter fetch gestartet wird.

Ohne das FlagKommt die alte Antwort ("Windkraft") später zurück als die neue ("Photovoltaik") — z.B. weil das Backend gerade unter Last war — würde sie den State mit Windkraft-Anlagen überschreiben, obwohl der Filter längst auf Photovoltaik steht. Die Tabelle würde für einen Moment falsche Daten zum aktuell gewählten Filter zeigen. Mit dem Flag: die Cleanup-Funktion des alten Effekt-Durchlaufs setzt cancelled = true, bevor der neue startet — die verspätete Windkraft-Antwort passiert ihr if (!cancelled) nicht mehr und wird stillschweigend verworfen.

Schritt für Schritt: _build_assets()

Ein eigener Zufallsgenerator mit festem Seed 42 — nicht das globale random-Modul. 2
Mehrtägiges "Wettermuster"

day_index = ts.timestamp() / 86400 rechnet den Zeitstempel in Tage seit 1970 um (mit Nachkommastellen für die Uhrzeit). weather = 1.0 + 0.9 * sin(2π · day_index / 4.2 + phase) erzeugt daraus eine Sinuswelle mit ca. 4,2 Tagen Periode — simuliert mehrtägige Schönwetter-/Sturmphasen, statt dass der Wind von einer Stunde auf die nächste beliebig springt.

3
Tagesgang + kurzfristiges Rauschen

diurnal macht Abend/Nacht leicht windiger als Mittag. ten_min_bucket = int(ts.timestamp() // 600) rundet den Zeitstempel auf 10-Minuten-Fenster ab — innerhalb eines Fensters liefert _stable_random denselben Rausch-Wert (die Böigkeit "springt" nicht bei jeder Sekunde), zwischen Fenstern ändert sie sich.

4
Alles multiplizieren, auf 2 Nachkommastellen runden

speed = base * weather * diurnal * noise — vier unabhängige, aber jeweils deterministische Faktoren ergeben zusammen einen Wert, der auf den ersten Blick "zufällig" wirkt, aber bei jeder erneuten Abfrage desselben Zeitpunkts exakt reproduzierbar ist.

Schritt für Schritt: wind_power_kw() — die Leistungskurve
def wind_power_kw(asset: Asset, ws: float) -> float:
    if ws < asset.cut_in_ms or ws >= asset.cut_out_ms:
        return 0.0                                    # zu wenig oder zu viel Wind -> Anlage steht
    if ws >= asset.rated_ws_ms:
        return asset.rated_power_kw                    # Nennwind erreicht -> volle Leistung, deckelt nach oben
    fraction = (ws - asset.cut_in_ms) / (asset.rated_ws_ms - asset.cut_in_ms)
    return round(asset.rated_power_kw * fraction**3, 1) # dazwischen: kubisch, nicht linear

Die dritte Potenz (fraction**3) ist kein Zufall: die Leistung einer Windturbine hängt physikalisch von der kinetischen Energie im Wind ab, die proportional zur dritten Potenz der Windgeschwindigkeit wächst — eine Verdopplung des Winds (innerhalb des Teillastbereichs) verachtfacht theoretisch die verfügbare Leistung. Das ist der Grund, warum die Kurve zwischen Einschalt- und Nennwind so viel steiler verläuft als eine simple lineare Rampe es täte.

Schritt für Schritt: _sync_alarm() — die zentrale Open/Close-Zustandsmaschine
def _sync_alarm(session, asset_id, code, severity, message, source, ts, condition_true):
    existing = session.execute(
        select(Alarm).where(Alarm.asset_id == asset_id, Alarm.code == code,
                             Alarm.source == source, Alarm.status == "active")
    ).scalar_one_or_none()          # 1: gibt es für GENAU diese Kombination schon einen offenen Alarm?

    if condition_true:
        if existing is None:
            session.add(Alarm(..., status="active", opened_at=ts))  # 2a: neu öffnen
        # 2b: existiert schon einer -> nichts tun, kein zweiter Alarm für dasselbe Problem
    else:
        if existing is not None:
            existing.status = "resolved"; existing.closed_at = ts   # 3: Bedingung nicht mehr wahr -> schließen

scalar_one_or_none() ist wichtig: es erwartet höchstens eine passende Zeile und gibt None zurück, falls keine existiert — würde man stattdessen .first() nutzen und es gäbe (durch einen Bug) versehentlich zwei offene Alarme derselben Kombination, würde das still das falsche Verhalten verschleiern statt es sichtbar zu machen.

Diese eine Funktion wird für alle Alarmarten wiederverwendet (SCADA-Störcodes, Minderleistung, SoC-Grenzen, Kommunikationsausfall) — jeweils nur mit einem anderen code und einer anderen condition_true-Berechnung aufgerufen. Das ist derselbe "eine Funktion, viele Aufrufer mit unterschiedlichen Parametern"-Gedanke wie bei der Dict-Lookup-Tabelle in normalize.py.

Schritt für Schritt: _integrate_trapezoidal() — die Mathematik dahinter
for (t0, v0), (t1, v1) in zip(points, points[1:]):
    dt_h = (t1 - t0).total_seconds() / 3600     # Zeitabstand in Stunden
    if 0 < dt_h <= MAX_GAP_HOURS:
        energy += (v0 + v1) / 2 * dt_h            # Trapezfläche = Durchschnittsleistung × Zeit

Leistung (kW) ist eine Momentaufnahme, Energie (kWh) ist Leistung über Zeit integriert — geometrisch die Fläche unter der Leistungskurve. Zwischen zwei Messpunkten kennt man die Kurve dazwischen nicht exakt, daher die Näherung: die Fläche eines Trapezes mit den beiden Messwerten als parallele Seiten und dem Zeitabstand als Höhe, (v0 + v1) / 2 * dt_h — der Mittelwert der beiden Leistungen mal die Zeit dazwischen. MAX_GAP_HOURS = 3 verhindert, dass eine große Lücke (z.B. genau an der Grenze zwischen Backfill-Ende und erstem Live-Poll) fälschlich als "3 Stunden am Stück mit dieser Leistung gelaufen" gewertet wird.

Warum performance_ratio_pv() Einstrahlung UND Leistung braucht
actual_energy_kwh = _integrate_trapezoidal(power_points)
irradiance_kwh_per_m2 = _integrate_trapezoidal(irradiance_points) / 1000.0
reference_yield_kwh = irradiance_kwh_per_m2 * asset.capacity_kwp
return actual_energy_kwh / reference_yield_kwh

Performance Ratio ist die Standard-Kennzahl, um eine PV-Anlage unabhängig vom Wetter zu bewerten: "wie viel Energie kam tatsächlich raus, im Verhältnis zu dem, was bei dieser Einstrahlung unter Idealbedingungen (STC-Nennleistung) rauskommen müsste?" Zwei separate Trapez-Integrationen (eine für Leistung, eine für Einstrahlung) über denselben Zeitraum, dann ins Verhältnis gesetzt.

export function parseUtc(iso: string): Date {
  const withZ = iso.endsWith("Z") || /[+-]\d{2}:\d{2}$/.test(iso) ? iso : `${iso}Z`;
  return new Date(withZ);
}

SQLite kennt keine echten zeitzonen-bewussten Timestamps — ein aus der DB gelesener Zeitstempel kommt als "naiver" ISO-String ohne Z oder Offset an (z.B. "2026-07-13T14:30:00"), obwohl er intern durchgehend UTC ist. new Date("2026-07-13T14:30:00") würde JavaScript aber als lokale Zeit interpretieren — auf einem Rechner in Wien (UTC+2) wäre das plötzlich zwei Stunden verschoben. Die Regex prüft, ob der String schon ein Z oder ein explizites Offset (+02:00) hat; falls nicht, wird Z angehängt, bevor new Date() ihn interpretiert.

StatCard: die simpelste Komponente — Label, Wert, optionales Sublabel/Akzentfarbe. Wird für praktisch jede Kennzahl im ganzen Projekt wiederverwendet, von "Ertrag heute" bis "Performance Ratio".

Ein Wert, der ganze Weg

Windgeschwindigkeit einer WEA — von der Simulation bis zum Chart im Browser

1
simulation.py: wind_speed_ms(asset, ts)

Berechnet deterministisch aus Standort-Basiswind, mehrtägigem Wettermuster und Böigkeits-Rauschen einen Wert wie 7.3 m/s für genau diesen Zeitpunkt — siehe die Schritt-für-Schritt-Erklärung weiter oben bei sim-simulation.

2
scada-simulator/main.py: read_live()

Ruft tag_values() für alle Anlagen auf und liefert eine flache Liste von TagReading-Objekten über GET /scada/live aus — {"asset_id": "wf-parndorf-wtg-07", "tag_name": "wind_speed_ms", "value": 7.3, "quality": "GOOD"}.

3
integration-service/scada_client.py: fetch_live_readings()

Holt diese Liste per async with httpx.AsyncClient(...) ab — ohne zu wissen, was ein "Windrad" ist, nur "irgendein JSON von der SCADA-URL".

4
ingestion.py: poll_once()

Reicht die Rohliste an normalize_tag_readings() weiter.

5
normalize.py

"wind_speed_ms" steht 1:1 in RAW_TAG_TO_METRIC (anders als active_power_kw, das auf power_kw umbenannt wird) — wird zu einer normalisierten Zeile {asset_id, metric: "wind_speed_ms", value: 7.3, timestamp}.

6
ingestion.py: _upsert_readings()

Schreibt die Zeile idempotent (ON CONFLICT DO NOTHING) in die readings-Tabelle. Gleichzeitig prüft evaluate_derived_alarms() in alarms.py, ob "viel Wind, aber wenig Leistung" zutrifft — dafür wird derselbe Wert 7.3 nochmal aus latest_by_asset gelesen, diesmal um die erwartete Leistung zu schätzen und mit der gemeldeten zu vergleichen.

7
frontend: AssetDetailPage.tsx

Ruft alle 15 Sekunden api.getAssetHistory(assetId, "wind_speed_ms", 24) über usePolling auf — ausgelöst entweder automatisch durch den Intervall-Timer oder sofort neu, wenn der Nutzer im Metrik-Umschalter auf "Windgeschwindigkeit" klickt (weil sich metric in den deps ändert).

8
integration-service/main.py: get_asset_history()

Fragt readings gefiltert nach asset_id, metric="wind_speed_ms" und Zeitfenster ab, gibt eine Liste {timestamp, value} zurück — als reines HistoryPointOut-Pydantic-Schema, nicht als rohes SQLAlchemy-Objekt.

9
frontend: HistoryChart.tsx

Bekommt die Liste als Props, parseUtc() korrigiert die Zeitzone jedes Zeitstempels, Recharts zeichnet die Linie — fertig ist der Chart, den der Techniker im Browser sieht, sieben Übersetzungsschritte nach der ursprünglichen Berechnung in simulation.py.
