Sprache & Typisierung

Type Hints + from __future__ import annotations

Jede Funktion ist vollständig typisiert (def wind_power_kw(asset: Asset, ws: float) -> float). Der __future__-Import lässt Annotationen als Text statt sofort ausgewertet — dadurch kann eine Klasse sich selbst oder eine später definierte Klasse referenzieren, ohne Reihenfolge-Probleme.

überall, z.B. simulation.py Zeile 1

@dataclass

Erzeugt automatisch __init__, __repr__, __eq__ aus den Feldern. field(default=0) statt = 0 direkt, weil dataclass sonst bei veränderlichen Defaults (Listen/Dicts) warnt.

scada-simulator/app/portfolio.py — Site, Asset

class AssetType(str, Enum)

Doppelt erbend: verhält sich wie ein String (JSON-serialisierbar, vergleichbar mit ==) und ist gleichzeitig ein typsicherer, abgeschlossener Wertebereich statt "irgendein String".

portfolio.py Zeile 18

Union-Typen mit |

float | str | None statt Optional[Union[float, str]] — seit Python 3.10 die kürzere Schreibweise für "eines von mehreren Typen oder nichts".

Modulweite Lookup-Tabellen (Dicts statt if/elif)

RAW_TAG_TO_METRIC ist ein Dict von rohem Tag-Namen auf einheitlichen Metrik-Namen — Erweiterung um einen neuen Tag heißt "eine Zeile ergänzen", nicht "eine if-Kette anfassen".

Async & I/O

async def / await

Der Prozess blockiert nicht, während er auf eine HTTP-Antwort vom Mock-SCADA wartet — in der Wartezeit kann derselbe Python-Prozess andere Anfragen (z.B. einen Dashboard-Request) bedienen.

scada_client.py, ingestion.py

Async Context Manager (async with)

async with httpx.AsyncClient(...) as client: öffnet die HTTP-Verbindung und garantiert, dass sie wieder geschlossen wird — auch wenn unterwegs ein Fehler auftritt.

scada_client.py — alle drei Funktionen

Endlosschleife mit asyncio.sleep

while True: await poll_once(); await asyncio.sleep(15) — der einfachste Weg, einen wiederkehrenden Hintergrund-Job zu bauen, ohne einen externen Scheduler (Cron, Celery) zu brauchen.

ingestion.py — poll_live_forever()

@asynccontextmanager für den App-Lebenszyklus

Der Code vor yield läuft beim Start (Tabellen anlegen, Backfill, Poll-Task starten), der Code nach yield beim Herunterfahren (Task abbrechen) — FastAPIs Weg für Start-/Stop-Hooks.

integration-service/app/main.py — lifespan()

Backend-Frameworks (FastAPI · SQLAlchemy · Pydantic)

Pydantic BaseModel

Deklarative Datenklassen, die FastAPI automatisch für Validierung, JSON-Serialisierung und die Swagger-Dokumentation nutzt — ein Schema, drei Zwecke.

schemas.py, scada-simulator/app/models.py

SQLAlchemy 2.0 Declarative Mapping

Mapped[str] + mapped_column(...) verbindet eine Python-Klasse 1:1 mit einer DB-Tabelle, inklusive Typprüfung durch den Editor/Typchecker — Asset.lat ist garantiert ein float, nicht "irgendwas aus der DB".

integration-service/app/models.py

Decorator-Routing (@app.get(...))

Die Funktion darunter wird nicht verändert, aber bei FastAPI als Handler für genau diesen Pfad registriert — deklarativ statt über eine zentrale Routing-Tabelle.

alle main.py

Session-per-Request (Unit of Work)

Jeder Request/Job holt sich am Anfang get_session() bzw. SessionLocal() und schließt sie am Ende in finally: session.close() — keine geteilte, langlebige DB-Verbindung.

integration-service/app/main.py, ingestion.py

Idempotentes Schreiben (ON CONFLICT DO NOTHING)

Backfill und Live-Poll können sich zeitlich überlappen — der Unique-Constraint auf (asset_id, metric, timestamp) plus on_conflict_do_nothing verhindert Duplikate, statt vorher mühsam zu prüfen "gibt's das schon?".

ingestion.py — _upsert_readings()

Deterministischer Zufall über Hashing

sha256(asset_id + zeitpunkt) als "Zufallszahl" — dieselbe Anlage + derselbe Zeitpunkt ergeben immer denselben Wert, ganz ohne gespeicherten Zustand. Das ist, was Live- und History-Abfrage konsistent macht.

simulation.py — _stable_random()

TypeScript / React

Interfaces als Vertrag mit dem Backend

Jedes interface in api/client.ts spiegelt eine Pydantic-Klasse aus schemas.py von Hand — ändert sich ein Feld im Backend, muss man es hier manuell nachziehen (kein generierter Client in dieser Demo).

frontend/src/api/client.ts

Generische Funktion get<T>()

Ein einziger Fetch-Wrapper für alle Endpunkte — der Aufrufer bestimmt über <T>, als welcher Typ die Antwort behandelt wird, ohne den Wrapper zu duplizieren.

api/client.ts Zeile 7

Custom Hook usePolling

Kapselt "sofort laden, dann alle N ms erneut, beim Unmount aufräumen" an einer Stelle — jede Seite ruft nur noch usePolling(fetcher, intervalMs) auf, statt useEffect+setInterval elfmal zu wiederholen.

lib/usePolling.ts

cancelled-Flag im useEffect-Cleanup

Verhindert, dass eine späte Antwort einer alten Anfrage (z.B. nach Filterwechsel) noch den State setzt, nachdem die Komponente längst neu geladen wurde — eine klassische React-Race-Condition-Falle.

hashlib.sha256Trigonometrie-ModellTuple-Rückgabe
def _stable_random(*parts: object) -> float:
    key = "|".join(str(p) for p in parts).encode()
    digest = hashlib.sha256(key).hexdigest()
    return int(digest[:12], 16) / 16**12

Ein Wert, der ganze Weg

Windgeschwindigkeit einer WEA — von der Simulation bis zum Chart im Browser

1
simulation.py: wind_speed_ms(asset, ts)

Berechnet deterministisch aus Standort-Basiswind, mehrtägigem "Wettermuster" und Böigkeits-Rauschen einen Wert wie 7.3 m/s für genau diesen Zeitpunkt.

2
scada-simulator/main.py: read_live()

Ruft tag_values() für alle Anlagen auf und liefert eine flache Liste von TagReading-Objekten über GET /scada/live aus — {"asset_id": "wf-parndorf-wtg-07", "tag_name": "wind_speed_ms", "value": 7.3, ...}.

3
integration-service/scada_client.py: fetch_live_readings()

Holt diese Liste per HTTP ab — ohne zu wissen, was ein "Windrad" ist, nur "irgendein JSON von der SCADA-URL".

4
ingestion.py: poll_once()

Reicht die Rohliste an normalize_tag_readings() weiter.

5
normalize.py

"wind_speed_ms" steht 1:1 in RAW_TAG_TO_METRIC (anders als active_power_kw, das auf power_kw umbenannt wird) — wird zu einer normalisierten Zeile {asset_id, metric: "wind_speed_ms", value: 7.3, timestamp}.

6
ingestion.py: _upsert_readings()

Schreibt die Zeile idempotent in die readings-Tabelle. Gleichzeitig prüft evaluate_derived_alarms() in alarms.py, ob "viel Wind, aber wenig Leistung" zutrifft (UNDERPERFORMANCE-Alarm).

7
frontend: AssetDetailPage.tsx

Ruft alle 15 Sekunden api.getAssetHistory(assetId, "wind_speed_ms", 24) über usePolling auf.

8
integration-service/main.py: get_asset_history()

Fragt readings gefiltert nach asset_id, metric="wind_speed_ms" und Zeitfenster ab, gibt eine Liste {timestamp, value} zurück.

9
frontend: HistoryChart.tsx

Bekommt die Liste als Props, parseUtc() korrigiert die Zeitzone, Recharts zeichnet die Linie — fertig ist der Chart, den der Techniker sieht.
